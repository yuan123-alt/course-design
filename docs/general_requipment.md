# 数据结构课程设计总体要求

---

## 一、总体总则

### 1.1 课程性质

本次课程设计为《数据结构》**综合性实践环节**，采用**三人一组**协作完成同一选题，实行**分组开发、分工负责、独立考核**模式。

### 1.2 分组要求

- 小组整体完成系统设计、编码、测试与文档编写
- 每位组员需明确个人负责模块，保证任务落地、工作量均衡
- **严禁**：全程一人包揽、搭便车、抄袭代码与文档

### 1.3 技术要求

所有选题要求**综合运用**以下核心知识，结合业务场景合理选用数据结构与算法，体现不同结构的特性与适用场景：

- 线性表、栈、队列、链表
- 树、二叉树、图
- 哈希、排序、查找
- 字符串算法

### 1.4 质量要求

整体程序需满足以下四项标准：

| 标准 | 说明 |
|------|------|
| **完整性** | 完整实现题目全部基础功能 |
| **健壮性** | 异常输入不崩溃，边界数据有兼容 |
| **实用性** | 贴近实际业务场景，操作流程合理 |
| **可扩展性** | 鼓励拓展附加功能、优化核心算法 |

### 1.5 架构要求

统一采用**模块化设计**，模块边界清晰、接口规范，便于多人协同开发。

### 1.6 协作要求

小组内部需做好沟通对接，保证各模块**数据互通、逻辑联动**，最终整合为一套完整可运行的系统。

---

## 二、开发环境与编程语言

### 2.1 开发环境

统一使用以下 IDE 完成代码编写、编译、调试、运行：

- **Dev-C++**
- **VSCode**

### 2.2 编程语言

- 使用标准 **C/C++** 开发
- **仅使用语言标准库**，禁止依赖第三方图形库、拓展类库

### 2.3 运行形式

统一为**控制台应用程序**，不做图形界面开发。

---

## 三、小组角色划分与分工标准

每组固定设置**三大角色**，三人一人一岗，角色不可互换代做，任务工作量基本均衡，覆盖设计、编码、测试、文档全流程。

### 3.1 角色一：系统设计 & 架构负责人（组员 A）

#### 核心负责内容

| 序号 | 职责 |
|------|------|
| 1 | 整体需求分析、可行性分析，梳理业务流程与数据流转规则 |
| 2 | 确定全局数据结构、结构体/结点定义、全局变量、模块划分与函数接口 |
| 3 | 绘制整体程序流程图、模块调用图、数据结构图，制定统一编码规范、命名规则 |
| 4 | 搭建程序主框架、主菜单、公共基础函数、数据初始化模块、文件读写公共接口 |
| 5 | 统筹小组进度，协调各模块对接问题，解决跨模块数据交互冲突 |

#### 报告负责部分

- 需求分析
- 系统概要设计
- 数据结构选型
- 整体架构
- 流程图绘制

---

### 3.2 角色二：核心功能 & 算法开发负责人（组员 B）

#### 核心负责内容

| 序号 | 职责 |
|------|------|
| 1 | 完成题目核心业务模块编码实现（如增删改查、排序、查找、队列/栈调度、树/图算法、KMP、最短路径、最小生成树等） |
| 2 | **手写实现**题目要求的各类经典算法，保证算法逻辑正确、运行高效 |
| 3 | 实现多数据结构联用、核心业务逻辑、状态流转、优先级调度等核心代码 |
| 4 | 对核心模块进行单元调试，修复模块内部语法与逻辑错误 |

#### 报告负责部分

- 详细设计
- 核心算法伪代码
- 核心模块实现思路
- 算法复杂度分析

---

### 3.3 角色三：交互、异常处理、测试 & 文档汇总负责人（组员 C）

#### 核心负责内容

| 序号 | 职责 |
|------|------|
| 1 | 完善人机交互界面、多级子菜单、操作提示、结果格式化输出 |
| 2 | 全局异常处理、容错逻辑开发：非法输入校验、业务异常拦截、边界数据处理 |
| 3 | 全系统联调、功能测试，设计合法/非法/边界测试用例，记录 BUG 并配合组员修复 |
| 4 | 日志记录、数据统计、可视化展示、数据持久化补充优化 |
| 5 | 整合全体代码，生成最终完整可运行程序；汇总所有资料，完成课程设计报告排版、整合、附录整理 |

#### 报告负责部分

- 测试用例
- 测试过程与结果
- 程序优缺点分析
- 优化方案
- 源码附录

---

### 3.4 补充说明

1. **工作量均等**：三人工作量要求均等，每人独立完成至少 **1/3 代码量 + 对应文档内容**
2. **归属标注**：公共基础部分由架构负责人完成，其余模块严格按角色划分归属人，**代码头部必须标注作者、负责模块**
3. **禁止代做**：允许组员之间技术交流、问题讨论，但**禁止代写代码、代写文档**

---

## 四、代码编写规范（含分组标注要求）

### 4.1 统一规范

全组执行同一套命名规则、缩进格式、括号风格：

- 变量、结构体、函数命名**见名知意**
- **禁止**无意义命名（如 `a`、`b`、`x1`、`temp` 等）

### 4.2 注释要求

#### 必须添加标准注释的位置

- 结构体定义
- 全局函数声明
- 核心算法实现

#### 每个功能函数 / 代码块头部必须标注

```
模块名称：
编写人：
编写时间：
```

以此明确代码归属。

#### 关键逻辑注释

关键逻辑、判断分支添加**行内注释**，说明处理意图。

### 4.3 数据结构约束

- 严格按照题目指定结构实现，**不得随意替换**
- 动态链表、动态结点必须**手动释放内存**，杜绝内存泄漏

### 4.4 模块接口

- 架构负责人定义统一接口
- 各开发人员严格按照接口规范编写代码
- 保证模块可**无缝拼接**

### 4.5 版本管理

小组做好代码备份，分模块保存源码，避免：

- 代码丢失
- 覆盖冲突

---

## 五、功能与逻辑实现要求

### 5.1 功能完整性

- 小组整体必须实现题目**全部基础功能**
- 核心模块归属到人，对应模块问题由负责组员主要整改

### 5.2 算法要求

排序、查找、遍历、图/树运算、字符串匹配等经典算法必须**手写实现**，**禁止调用系统内置函数替代**。

### 5.3 业务逻辑

- 严格遵循题目业务规则、数据联动、状态流转、排队/调度逻辑
- 多模块联动处需**提前对接**

### 5.4 模块独立性

各功能模块**高内聚、低耦合**，单一模块故障不影响整体框架运行。

---

## 六、人机交互、异常处理要求

### 6.1 交互界面

- 菜单层级清晰
- 操作提示简洁易懂
- 输出格式整齐美观
- **该部分由组员 C 主导实现与优化**

### 6.2 容错处理

全面拦截以下异常场景，程序**不得崩溃、闪退、死循环**：

- 非法输入
- 数据不存在
- 库存/名额不足
- 文件读写失败
- 重复操作

### 6.3 边界数据兼容

以下边界数据需做兼容处理：

- 空数据
- 数据已满
- 极值

---

## 七、文件操作与数据持久化要求

### 7.1 文件格式

涉及文件功能的题目，统一实现 **TXT 文件**导入、保存、开机自动加载；文件格式全组统一。

### 7.2 分工

| 角色 | 职责 |
|------|------|
| **架构负责人（组员 A）** | 搭建文件基础接口 |
| **对应开发人员** | 实现业务层文件调用 |
| **组员 C** | 统一完善文件异常捕获 |

### 7.3 文件分类

区分不同类型数据文件，分类存储，便于维护与测试。

---

## 八、程序测试要求（分组执行）

### 8.1 单元测试

各模块负责人先完成自身模块单元测试，确保单独运行无误。

### 8.2 整体联调

由**组员 C** 主导全程序集成测试，设计多组测试用例：

- 合法数据测试
- 非法数据测试
- 边界数据测试

### 8.3 问题整改

| 步骤 | 说明 |
|------|------|
| 发现 BUG | 测试发现 BUG 后，对应模块负责人限期修复 |
| 复测验证 | 全体组员配合复测 |
| 记录归档 | 在报告中记录问题、原因、解决方案 |

### 8.4 性能测试

核心算法需统计并分析以下指标（由算法负责人完成）：

- 算法耗时
- 比较次数
- 冲突次数

---

## 十一、答辩要求（分组答辩、逐个提问、单独评分）

### 11.1 答辩形式

- 小组**集体上台**演示程序
- **分人讲解**各自负责部分
- 每人必须**独立讲解、独立作答**

### 11.2 演示要求

完整展示系统所有功能，由小组配合完成操作演示。

### 11.3 个人答辩内容

| 角色 | 讲解内容 | 回答方向 |
|------|----------|----------|
| **组员 A** | 整体架构、数据结构选型、模块划分、框架设计思路 | 架构、数据结构相关问题 |
| **组员 B** | 核心算法、业务逻辑、代码实现思路 | 算法、功能逻辑相关问题 |
| **组员 C** | 测试过程、异常处理、程序优缺点、优化方案 | 测试、交互、文档相关问题 |

### 11.4 考核规则

- 整体看**系统完成度**，个人看**对应模块掌握程度、代码理解、现场答题情况**
- 实行「**小组分 + 个人分**」综合评定
- 个人未完成自身任务、无法讲解负责模块，按**个人不及格**处理

---

## 十二、纪律与考核说明

### 12.1 学术诚信

- **严禁**小组之间互相抄袭代码、照搬报告
- **严禁**组内一人代做全部任务
- 一经查实，**全组按不及格处理**

### 12.2 进度要求

- 按实习计划分阶段完成任务
- 每日同步模块进度
- 逾期未完成对应模块，追究对应负责人责任

### 12.3 成绩构成

| 评分项 | 占比 | 说明 |
|--------|------|------|
| **程序整体质量** | **40%** | 含功能完整性、代码规范、健壮性（结合个人模块完成度打分） |
| **课程设计报告** | **30%** | 结合个人撰写章节质量、分工合理性打分 |
| **现场答辩** | **30%** | 按个人讲解、答题表现单独打分 |

### 12.4 责任判定

若出现模块衔接失败、功能缺失：

- 优先判定**对应模块负责人**责任
- 小组共同配合整改
